1. Introduction and Software Requirements

1.1 Project Overview

Roamly is a community-driven mobile travel application built for solo riders, bikers, and backpackers. Its core purpose is to make solo travel safer and more social by allowing users to discover hidden gems — user-submitted points of interest — on a live map, and to connect with nearby travellers for companionship. The application runs as a free, zero-cost prototype on the Firebase Spark plan, meaning every feature is designed to work within the free tier of Google's cloud services.

The application adopts a login-first architecture with role-based routing: regular users land on the interactive map home screen, while the admin account is redirected to a moderation dashboard used to approve or reject crowd-sourced spot submissions.

1.2 Objectives

  • Enable users to sign up, sign in (email/password or Google OAuth), and manage a profile.
  • Display an interactive, tile-based map powered by Mapbox showing approved spots.
  • Allow users to submit new spots at their current GPS location or at an arbitrary map-picked coordinate.
  • Surface a Find Companion screen where discoverable travellers appear as real-time markers.
  • Provide an Admin Dashboard to moderate pending spot submissions (approve or reject).
  • Keep all backend costs at zero by using Firebase's free tier and Mapbox's free geocoding quota.

1.3 Software Requirements Specification (SRS)

Functional Requirements

  • FR-01 — The system shall allow new users to register with name, email, phone number, and password.
  • FR-02 — The system shall support Google Sign-In as an alternative authentication method.
  • FR-03 — The system shall display approved spots as map markers; tapping a marker shows its details.
  • FR-04 — The system shall obtain and display the device's GPS coordinates as a blue dot on the map.
  • FR-05 — The system shall allow authenticated users to submit a new spot (name, description, type, tags, rating) either at their current location or at a map-picked coordinate.
  • FR-06 — All user-submitted spots shall start in pending status and only appear publicly upon admin approval.
  • FR-07 — The Admin account can add spots that are immediately approved and publicly visible.
  • FR-08 — Authenticated users can toggle a "discoverable" switch to broadcast their GPS coordinates to other users.
  • FR-09 — The Find Companion screen shall display discoverable users as map markers with their real-time location; tapping a marker offers a connection request.
  • FR-10 — The search bar shall query both the Mapbox Geocoding API and the local Firestore approved-spot collection, merging and ranking results.
  • FR-11 — The system shall support dark mode and light mode toggling, persisted across sessions.

Non-Functional Requirements

  • NFR-01 Performance — Map tiles shall render within 2 seconds on a standard 4G connection.
  • NFR-02 Security — Firebase API keys shall be stored in an environment file (.env) loaded at runtime; the file is excluded from version control.
  • NFR-03 Usability — All primary user actions (login, view map, add spot, find companion) must be reachable within three taps from the home screen.
  • NFR-04 Reliability — All network calls shall be wrapped in try-catch blocks, with appropriate SnackBar notifications shown on failure.
  • NFR-05 Scalability — The location-update logic shall only save to Firestore when the user has moved more than 500 metres, minimising unnecessary write operations.

2. Design

2.1 Application Architecture

Roamly follows a feature-first directory layout inside lib/. Screens and widgets are grouped by feature (auth, home, profile, companions, admin), while shared infrastructure — services, themes, constants — lives in a core/ module. Data classes reside in models/. This enforces the Separation of Concerns principle so that UI code never calls the database directly. 

lib/
├── core/
│   ├── constants/       // AppConstants, MapboxConfig
│   ├── services/        // LocationService, MapboxGeocodingService, GoogleAuthService, UserService, RequestService
│   └── themes/          // AppTheme (light and dark), ThemeProvider
├── features/
│   ├── auth/screens/    // SplashScreen, LoginScreen (welcome, sign-in, sign-up flow)
│   ├── home/            // HomeScreen, AddSpotDialog, LocationSearchDelegate

2.2 UI Design

The visual language is deep-navy blue (#1A237E) with indigo accents, using the Poppins and Google Fonts typefaces for a clean, modern feel. The login screen renders a navy-to-indigo gradient with a custom CustomPainter topographic line overlay and animated AnimatedSwitcher transitions between the three sub-screens (Welcome, Sign In, and Sign Up).

The home screen is dominated by a full-bleed Mapbox-tiled map. Floating action buttons for My Location and Add Spot overlay the map at the bottom-right corner. Navigation is handled via a bottom navigation bar (Explore, Trips, Companions, and Profile) and a side drawer that fetches the signed-in user's profile photo and name in real time from Firestore via a StreamBuilder.

Both light and dark themes are supported. Theme preference is persisted using shared_preferences and applied via a global ValueNotifier<ThemeMode> without the need for a separate state-management library.

3. Implementation

3.1 Core Services

Three primary services handle the device and cloud integrations used throughout the application:

Geolocator (GPS)
geolocator: ^14.0.2
Requests the device's current GPS position and checks for runtime location permissions. Used by both the Home screen (blue-dot indicator) and the Find Companion screen (real-time location sharing).
Firebase / Cloud Firestore
cloud_firestore: ^6.1.2
NoSQL cloud database used as the primary data store. All users, spots (locations), and companion requests are persisted here. Real-time streams (snapshots()) power the companion map and the drawer's live user info.
Mapbox Geocoding API
http: ^1.2.0 + flutter_map: ^8.2.2
Two-stage Mapbox Search Box API (suggest followed by retrieve) provides forward geocoding for the in-app location search. Results are merged with Firestore spot results and ranked using a scoring algorithm that combines keyword matching, fuzzy similarity (Levenshtein distance), and proximity bias.

Snippet 1 — Firebase Initialisation and Environment Loading (main.dart)

Snippet 2 — GPS and Permission Handling (home_screen.dart)

The application checks each permission state explicitly and moves the map camera to the user's position on success.

Future<void> _handleLocationButton() async {
  bool serviceEnabled = await Geolocator.isLocationServiceEnabled();
  if (!serviceEnabled) {
    ScaffoldMessenger.of(context).showSnackBar(
      const SnackBar(content: Text('Location services are disabled.')),
    );
    return;
  }

  LocationPermission permission = await Geolocator.checkPermission();
  if (permission == LocationPermission.denied) {
    permission = await Geolocator.requestPermission();
  }

  final position = await Geolocator.getCurrentPosition();
  setState(() {
    _currentPosition = LatLng(position.latitude, position.longitude);
  });
  _mapController.move(_currentPosition!, 19);
}

Snippet 3 — Firestore Moderation: LocationService

Spots submitted by users are saved with status: "pending". Admin approval calls updateLocationStatus to flip the flag to "approved", making the spot visible on the public map. 

/// Add a user-submitted spot (always pending)
Future<String?> addLocation(LocationModel newLocation) async {
  try {
    final data = newLocation.toMap();
    data.remove('id');
    data['status'] = 'pending';                   // always pending for users
    data['createdAt'] = FieldValue.serverTimestamp();

    await _firestore.collection(_collection).add(data);
    return null;                                    // null = success
  } catch (e) {
    return 'Failed to add location: $e';
  }
}

/// Admin moderation: flip status field
Future<void> updateLocationStatus(String id, String status) async {
  await _firestore.collection(_collection)
      .doc(id)
      .update({'status': status});
}

Snippet 4 — Two-Stage Mapbox Search (mapbox_geocoding_service.dart)

The search uses a session token to keep suggest and retrieve calls in one billing session, keeping costs within Mapbox's free tier. A 100 ms rate limiter prevents API abuse on fast typing. 

Future<List<SearchResult>> getSuggestions(
  String query, {LatLng? proximity}
) async {
  if (query.trim().isEmpty) return [];

  await _enforceRateLimit();                    // min 100 ms between requests
  _sessionToken ??= _generateSessionToken();   // reuse session for cost savings

  final uri = Uri.parse('...mapbox.../suggest')
      .replace(queryParameters: {
        'q': query,
        'access_token': MapboxConfig.accessToken,
        'session_token': _sessionToken!,
        'types': 'poi,address',
        if (proximity != null)
          'proximity': '${proximity.longitude},${proximity.latitude}',
      });

  final response = await http.get(uri);
  if (response.statusCode == 200) {
    final data = jsonDecode(response.body);
    return (data['suggestions'] as List)
        .map((s) => SearchResult.fromMapboxSuggestion(s, 1.0))
        .toList();
  }
  return [];
}

Snippet 5 — Real-Time Companion Discovery (user_service.dart)

getNearbyUsers() returns a live Stream from Firestore, automatically updating the map whenever another user goes discoverable or changes position. 

Stream<List<UserProfile>> getNearbyUsers() {
  return _firestore
      .collection('users')
      .where('isDiscoverable', isEqualTo: true)
      .snapshots()
      .map((snapshot) {
        final currentUid = _auth.currentUser?.uid;
        return snapshot.docs
            .map((doc) => UserProfile.fromMap(doc.data()))
            .where((user) => user.uid != currentUid && user.location != null)
            .toList();
      });
}

Snippet 6 — Email/Password Sign-Up with Firestore Profile Creation (login_screen.dart)

On a successful Firebase Auth registration, the user's additional profile details (name, phone) are immediately written to Firestore under the same UID. 

Future<void> _handleSignup() async {
  if (!_signupFormKey.currentState!.validate()) return;

  final credential = await FirebaseAuth.instance
      .createUserWithEmailAndPassword(
        email: _signupEmailCtrl.text.trim(),
        password: _signupPasswordCtrl.text,
      );

  final userProfile = UserProfile(
    uid: credential.user!.uid,
    email: _signupEmailCtrl.text.trim(),
    name: _signupNameCtrl.text.trim(),
    phoneNumber: _signupPhoneCtrl.text.trim(),
    createdAt: DateTime.now(),
  );

  await FirebaseFirestore.instance
      .collection('users')
      .doc(credential.user!.uid)
      .set(userProfile.toMap());

  Navigator.pushReplacement(context,
      MaterialPageRoute(builder: (_) => const HomeScreen()));
}
